Scripted Verification: Automated Assurance in African Infrastructure Builds

The simplest test is not the most powerful. The most powerful test is one that can be run every time, by anyone, in the same conditions that produce a pass or a fail in production. This is not a trivial point. Most of what is called "QA" on African software projects amounts to a human looking at a page and saying "looks good." That judgment expires at the next commit. It is not infrastructure. It is an anecdote.

On May 19, the Afriplan Global Solution live site surfaced a discrepancy. Eight mineral spans — Beryl, Tungsten, Gold, Lithium, Cassiterite, Coltan, Copper, Smoky Quartz — were rendering correctly in the live browser, but an automated script was flagging them as absent. The disconnect was between what a DOM inspector measured (static structure) and what the browser was actually injecting (innerHTML via apply()). The test was technically correct on static structure but wrong on rendered output. The error was not in the test. It was in the thing the test was measuring.

The Scripted Verification Loop

What emerged was not a single bug fix. It was a design pattern with four properties:

  • Reproducible. Every condition is a single query against the live page — no fuzzy matching, no "looks good to me." Pass or fail is a binary property of the live DOM at query time.
  • Idempotent. Running the same query twice, against the same live state, produces identical output. If output changes, something genuinely changed.
  • Read, not assert. The script extracts actual rendered state. It does not load expectations from a manual checklist. Verification reads what exists rather than testing against what a human assumed would be there.
  • Writable reference output. The script writes a verification record — a timestamped capture of every check result. That record is itself the artifact.
A verification system that requires human approval before producing evidence is not verification. It is a ceremony. The real check happens when no one is looking.

Why This Architecture Matters for African Infrastructure

The Playwright step was effective, but the pattern is portable. Every African startup, government digital service, or open-source project could benefit from verification loops built this way. Automate the checks that already matter most. Run them each time something changes. Capture the result without waiting for a human to read it.

Vérification scriptée : Assurance automatisée dans les constructions d'infrastructure africaine

Le test le plus simple n'est pas le plus puissant. Le test le plus puissant est celui qui peut être exécuté à chaque fois, par n'importe qui, dans les mêmes conditions qui produisent un succès ou un échec en production. Ce point n'est pas trivial. La plupart de ce qui est appelé "assurance qualité" dans les projets logiciels africains se réduit à un humain qui regarde une page et dit "ça a l'air bon." Ce jugement expire à la prochaine modification. Ce n'est pas de l'infrastructure. C'est une anecdote.

Le 19 mai, le site d'Afriplan Global Solution en direct a fait apparaître une divergence. Huit spans minéraux — Beryl, Tungstène, Or, Lithium, Cassitérite, Coltan, Cuivre, Quartz Fumé — se rendaient correctement dans le navigateur en direct, mais un script automatisé les signalait comme absents. Le découplage se situait entre ce que mesurait l'inspecteur DOM (structure statique) et ce que le navigateur injectait réellement (innerHTML via apply()). Le test était techniquement correct sur la structure statique mais erroné sur la sortie rendue. L'erreur n'était pas dans le test. Elle était dans ce que le test mesurait.

La boucle de vérification scriptée

Ce qui a émergé n'était pas une correction de bug isolée. C'était un motif de conception doté de quatre propriétés :

  • Reproductible. Chaque condition est une requête unique sur la page en direct — pas de correspondance floue, pas de "ça a l'air bon." Succès ou échec est une propriété binaire du DOM en direct au moment de la requête.
  • Idempotent. Exécuter la même requête deux fois contre le même état en direct produit une sortie identique. Si la sortie change, quelque chose a réellement changé.
  • Lecture, pas affirmation. Le script extrait l'état rendu réel. Il ne charge pas d'attentes d'une liste de contrôle manuelle. La vérification lit ce qui existe plutôt que tester contre ce qu'un humain supposait être présent.
  • Sortie de référence inscriptible. Le script écrit un enregistrement de vérification — une capture horodatée de chaque résultat. Cet enregistrement est lui-même le produit.
Un système de vérification qui nécessite l'approbation humaine avant de produire une preuve n'est pas une vérification. C'est une cérémonie. Le vrai contrôle se produit quand personne ne regarde.

Pourquoi cette architecture compte pour l'infrastructure africaine

L'étape Playwright a été efficace, mais le motif est portable. Chaque startup africaine, service numérique gouvernemental, ou projet open-source pourrait bénéficier de boucles de vérification construites ainsi. Automatiser les contrôles qui comptent déjà le plus. Les exécuter à chaque modification. Capturer le résultat sans attendre qu'un humain le lise.
